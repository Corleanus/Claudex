## Steps

### WP-01: Extract magic numbers to named constants (12 findings)

**Strategy**: Replace bare numeric literals (>1000) with named constants. Many are time-duration calculations (`24 * 60 * 60 * 1000`) — extract to descriptively named `const` declarations at module or block scope.

**Files and actions**:

1. **`src/lib/decay-engine.ts`** (lines 97, 224, 225) — `180 * 86400 * 1000` is already a constant `IMMUNITY_WINDOW_MS` at line 97 (false positive — the constant IS named). Lines 224-225: `86400000` (ms per day) used in day-since-access calculation → extract `const MS_PER_DAY = 86_400_000;` at module scope.

2. **`src/gsd/cross-phase-writer.ts`** (line 24) — `5 * 60 * 1000` is already `const DEBOUNCE_MS` with comment. False positive — already named. Resolve as wontfix.

3. **`src/db/audit.ts`** (line 223) — `retentionDays * 24 * 60 * 60 * 1000` → extract `const MS_PER_DAY = 24 * 60 * 60 * 1000;` at function scope or use a shared constant.

4. **`src/gsd/summary-writer.ts`** (line 28) — `5 * 60 * 1000` is already `const DEBOUNCE_MS` with comment. False positive — already named. Resolve as wontfix.

5. **`src/hooks/session-end.ts`** (line 188) — `5000` (stale lock threshold) → extract `const STALE_LOCK_MS = 5000;` near the lock logic.

6. **`src/hooks/user-prompt-submit.ts`** (lines 345, 37) — Line 37 is `DEFAULT_CONTEXT_TOKEN_BUDGET = 4000` (already named). Line 345: `5 * 60 * 1000` → extract `const METRICS_STALENESS_MS = 5 * 60 * 1000;` in the block.

7. **`src/lib/recovery.ts`** (lines 49-52) — Four constants are already named (`STALE_PORT_FILE_MS`, `STALE_COOLDOWN_MS`, `MAX_COOLDOWN_DURATION_MS`, `ORPHAN_SESSION_MS`). The expressions like `24 * 60 * 60 * 1000` are the RHS of named constants. False positives — already named. Resolve as wontfix.

8. **`src/lib/retention.ts`** (line 35) — `retentionDays * 24 * 60 * 60 * 1000` → same pattern as audit.ts. Extract `const MS_PER_DAY = 24 * 60 * 60 * 1000;` locally.

9. **`src/lib/token-gauge.ts`** (line 87) — `n >= 1000` in a formatting function → `const KILO_THRESHOLD = 1000;` locally, or mark wontfix (1000 is a well-known formatting boundary).

10. **`src/shared/epoch.ts`** (line 24-25) — `1e12` threshold. Already documented by comment explaining the heuristic. Mark wontfix — the comment is the documentation.

11. **`src/shared/logger.ts`** (line 19) — `5 * 1024 * 1024` is already `const MAX_LOG_SIZE`. False positive — already named. Resolve as wontfix.

12. **`src/wrapper/flush-trigger.ts`** (line 111) — `.slice(-10000)` → extract `const MAX_REASONING_CHARS = 10_000;`.

**Summary**: ~6 files need actual constant extraction, ~6 are already named constants (wontfix).

### WP-02: Remove non-null assertions (3 findings, 22 instances)

**Files and actions**:

1. **`src/shared/config.ts`** (17 instances, lines 47-99) — All `defaults.hologram!.enabled` etc. where `defaults` is `DEFAULT_CONFIG` which is a known-shape object. These are safe because `DEFAULT_CONFIG` always has these properties. Fix: declare `DEFAULT_CONFIG` with a non-optional type, or extract the sub-objects before the validation function so TypeScript knows they're defined: `const hDefaults = defaults.hologram!; ...` → one assertion per block instead of 17.

2. **`src/hooks/user-prompt-submit.ts`** (1 instance, line 151) — `gsdState!.position!.phase` inside a block guarded by `if (gsdState?.position)`. Safe, but TypeScript doesn't narrow after `.find()`. Fix: extract to a local variable with a guard: `const pos = gsdState?.position; if (pos) { ... pos.phase ... }`.

3. **`src/lib/context-assembler.ts`** (4 instances, lines 164-209) — `sources.hologram!.hot` etc. used after `const hasHologram = !!sources.hologram;`. Fix: extract `const hologram = sources.hologram;` at the top, then use `if (hologram) { ... hologram.hot ... }`.

### WP-03: Add logging to catch-return-default blocks (3 findings, 4 instances)

**Context**: These are `catch` blocks that return default/empty objects, which is intentional for "never throws" APIs. The smell is that errors are silently swallowed. In all 3 cases, logging already exists (`log.error` or `log.warn`). The tool flags them because the catch returns an object instead of re-throwing.

**Files and actions**:

1. **`src/lib/decay-engine.ts`** (line 265) — Already logs `log.error('Failed to prune observations:', err)` before returning `{ pruned: 0, remaining: 0 }`. This is correct behavior for a "never throws" API. Mark wontfix with rationale.

2. **`src/shared/health.ts`** (lines 83, 112) — Both already log via `log.warn()` before returning defaults. These are "never throws" health check functions — returning a degraded status is the correct behavior. Mark wontfix with rationale.

### WP-04: Fix console.error without throw/return (1 finding, 5 instances)

**File**: `src/gsd/phase-transition-cli.ts`

**Analysis**: This is a CLI entry point (`run()` function). The pattern is:
- Line 184: `console.error(USAGE)` in `printUsage()` — helper function, deliberately void
- Lines 193, 200: `console.error(...)` followed by `process.exit(1)` — process.exit IS the exit path
- Lines 239, 243: Same pattern — console.error + process.exit or within a catch that calls process.exit

**Action**: The `process.exit(1)` calls serve the same purpose as `throw` in a CLI context. Mark wontfix with rationale: "CLI entry point — process.exit(1) serves as the error exit path."

### WP-05: Add sort comparators (1 finding, 2 instances)

**File**: `src/gsd/cross-phase-writer.ts` (lines 254, 309)

**Analysis**: Both are `.sort()` on arrays of filename strings (`.md` files). Default lexicographic sort is the intended behavior (alphabetical/chronological filenames like `session-2026-02-27.md`).

**Action**: Add explicit comparator `(a, b) => a.localeCompare(b)` to both calls to make intent clear and satisfy the detector.

### WP-06: Add default case to switch (1 finding, 1 instance)

**File**: `src/gsd/phase-transition-cli.ts` (line 205)

**Analysis**: Switch on `parsed.event` which is a union type `'phase-start' | 'phase-end' | 'plan-complete'`. All three cases are handled. TypeScript exhaustiveness ensures coverage.

**Action**: Add `default: { const _exhaustive: never = parsed.event; throw new Error(\`Unexpected event: \${_exhaustive}\`); }` for exhaustiveness checking at both compile and runtime.

### WP-07: Mark monster function as wontfix (1 finding)

**File**: `src/checkpoint/writer.ts` line 230 — `writeCheckpoint()` is 151 lines.

**Analysis**: The function is a sequential pipeline: read inputs → build object → serialize → write → return. It's barely over the 150 LOC threshold. Splitting would create artificial helper functions with one caller each. The function is well-structured with clear comment sections.

**Action**: Mark wontfix with rationale: "Sequential pipeline, 151 LOC (1 line over threshold). Well-sectioned internally. Splitting would create helpers with single callers."

### WP-08: Mark dead function as wontfix (1 finding)

**File**: `src/hologram/protocol.ts` line 96 — `buildRequest()` flagged as "body is empty/return-only."

**Analysis**: The function constructs and returns a `SidecarRequest` object with a fresh UUID. It's actively imported and used in `launcher.ts` and `client.ts` (5+ call sites). The detector flags "return-only" bodies, but a factory function IS its return value — this is a false positive.

**Action**: Mark wontfix with rationale: "Factory function — return-only body is the correct pattern. Actively used in launcher.ts and client.ts."

### WP-09: Resolve signature variance (1 finding)

**Finding**: `estimateTokens()` has 2 parameter signatures across 3 files:
- `src/checkpoint/writer.ts:195` — `estimateTokens(checkpoint: Checkpoint): number` (takes object, serializes internally)
- `src/checkpoint/loader.ts:303` — `estimateTokens(text: string): number` (takes text)
- `src/lib/context-sections.ts:27` — `estimateTokens(text: string): number` (takes text, exported)

**Analysis**: The writer's version does `JSON.stringify` then divides by 4. The other two divide text length by 4. The writer version takes a different input type because it needs to serialize first — this is an intentional signature difference.

**Action**: Rename `writer.ts`'s version to `estimateCheckpointTokens()` to make the distinction explicit and remove the signature variance flag.
