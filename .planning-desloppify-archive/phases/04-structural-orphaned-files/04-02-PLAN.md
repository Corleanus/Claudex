# Plan 04-02: Resolve Orphaned File Findings

## Situation

54 files are flagged as orphaned ("zero importers, not an entry point"). However, this codebase is a **hook system** where:

1. **Entry points are hooks** — files in `src/hooks/` (excluding `_infrastructure.ts`) are esbuild entry points. They are never imported by other source files; they ARE the top of the dependency tree.
2. **`src/gsd/phase-transition-cli.ts`** is also an esbuild entry point (defined in `build.ts` line 14).
3. **Dynamic imports are invisible to the detector** — most non-hook files are imported via `await import('../path/to/module.js')` inside hook handlers. The orphan detector only tracks static `import` statements and misses dynamic imports entirely.

**53 of 54 orphaned findings are false positives.** Only `src/db/vectors.ts` is genuinely dead code.

### About `src/db/vectors.ts`

This is a 122-LOC `FTS5VectorStore` class that wraps the existing FTS5 search functions behind a uniform interface. It was designed as a future abstraction layer for swapping in real vector stores. It has:
- Zero importers (static or dynamic) anywhere in the codebase
- 1 test file (`tests/db/vectors.test.ts`) that imports from `search.ts`, not `vectors.ts` directly
- Its only export (`FTS5VectorStore`) was previously flagged as a dead export in Phase 3 and marked as a test-consumed false_positive (finding `exports::src/db/vectors.ts::FTS5VectorStore`)

Given that vectors.ts has no production consumers and no test consumers (the test imports search functions, not the vector store class), it is genuinely dead code. However, it was deliberately designed as a future abstraction. Decision: delete it. If a vector store abstraction is needed later, it can be recreated. Dead code that "might be useful someday" is still dead code.
