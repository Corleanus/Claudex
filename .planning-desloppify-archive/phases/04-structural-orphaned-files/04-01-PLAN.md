# Plan 04-01: Resolve Structural Findings (Decompose Large Files)

## Situation

7 files exceed the 500 LOC structural threshold. All are T3 (medium) findings.

| File | LOC | Section Count | Nature |
|------|-----|---------------|--------|
| `src/hooks/user-prompt-submit.ts` | 704 | 2 sections | Hook entry point: query helpers + main handler |
| `src/gsd/state-reader.ts` | 566 | 5 sections | Pure parsers + filesystem reader + cross-reference builder |
| `src/lib/context-assembler.ts` | 550 | 3 sections | ~12 section builders + 2 assembly modes |
| `src/checkpoint/loader.ts` | 547 | 5 sections | YAML helpers + validation + filesystem + formatting + public API |
| `src/db/search.ts` | 521 | 5 sections | 2 migration functions + 3 table-specific search + unified search |
| `src/hooks/session-start.ts` | 521 | 2 sections | Bootstrap/registration + main handler |
| `src/hooks/session-end.ts` | 502 | ~10 numbered sections in main handler |

### Assessment: Structural decomposition is high-risk for this codebase

These 7 files are the core of a **hook system where each file is a self-contained entry point** bundled by esbuild. Key considerations:

1. **Hook files are entry points, not libraries.** `session-start.ts`, `session-end.ts`, `user-prompt-submit.ts` are bundled into standalone `.mjs` files. Splitting them into multiple files means more cross-file imports, but the bundle is the same — the LOC in the source file doesn't affect runtime behavior.

2. **The files are already well-structured internally.** Each uses clearly demarcated `// ===` section headers. Functions are focused and self-documenting. The "largeness" comes from having many related functions in one domain, not from monolithic spaghetti.

3. **Splitting carries real risk.** 1207 tests pass today. Decomposing 7 files means moving functions, updating imports across the codebase and tests, potentially breaking the build. The benefit is a desloppify score improvement; the risk is regressions.

4. **The 500 LOC threshold is arbitrary.** Files at 502-550 LOC are barely over. Only `user-prompt-submit.ts` (704) is significantly over.

### Recommended approach: Split where natural seams exist, mark remainder as wontfix

For each file, analyze whether there's a **clean extraction** that produces two files each under 500 LOC and doesn't create unnecessary coupling. Where extraction is clean, do it. Where the file is a cohesive unit that's barely over threshold, mark it `wontfix`.

## Steps

### WP-01: Extract search migrations from `src/db/search.ts` (521 LOC)

**Rationale**: `search.ts` contains two migration functions (`migration_2` at line 34, `migration_4` at line 83) that are logically separate from the search query functions. Migrations are called once during DB setup; search functions are called at runtime. Clean seam.

**Action**:
1. Create `src/db/search-migrations.ts` with `migration_2()` and `migration_4()` (plus their helper types)
2. Update `src/db/migrations.ts` to import from `search-migrations.ts` instead of `search.ts`
3. Update all test imports (`tests/db/search.test.ts`, `tests/e2e/degradation-tiers.test.ts`, `tests/hologram/*.test.ts`, `tests/hooks/pre-compact.test.ts`) that import migration functions
4. Verify: `tsc --noEmit`, `vitest run`, `node build.ts`

**Expected result**: `search.ts` drops to ~380 LOC, `search-migrations.ts` is ~140 LOC.

### WP-02: Extract query helpers from `src/hooks/user-prompt-submit.ts` (704 LOC)

**Rationale**: The file has a clear two-part structure: (1) query/extraction helper functions (lines 48-245: `queryHologram`, `extractRecentFiles`, `extractKeywords`, `queryFts5`, `getRecent`) and (2) the main hook handler. The helpers are pure functions with no hook-specific state.

**Action**:
1. Create `src/hooks/_prompt-queries.ts` (prefixed with `_` so `build.ts` doesn't treat it as an entry point)
2. Move `queryHologram`, `extractRecentFiles`, `extractKeywords`, `queryFts5`, `getRecent` into the new file
3. Update `user-prompt-submit.ts` to import from `./_prompt-queries.js`
4. Update `tests/hooks/` if any test directly imports these helpers
5. Verify: `tsc --noEmit`, `vitest run`, `node build.ts`

**Expected result**: `user-prompt-submit.ts` drops to ~460 LOC, `_prompt-queries.ts` is ~240 LOC.

### WP-03: Extract section builders from `src/lib/context-assembler.ts` (550 LOC)

**Rationale**: The file has ~12 small `buildXxxSection()` functions (lines 65-203) that are pure string formatters. They don't depend on each other and are only called from the two assembly functions. Extracting them leaves the assembly logic compact.

**Action**:
1. Create `src/lib/context-sections.ts` with all `buildXxxSection` functions + `estimateTokens` + `formatTimeAgo` helpers
2. Update `context-assembler.ts` to import from `./context-sections.js`
3. Verify: `tsc --noEmit`, `vitest run`, `node build.ts`

**Expected result**: `context-assembler.ts` drops to ~250 LOC, `context-sections.ts` is ~300 LOC.

### WP-04: Extract checkpoint formatting from `src/checkpoint/loader.ts` (547 LOC)

**Rationale**: The file has two distinct responsibilities: (1) loading/recovery logic (YAML parsing, filesystem traversal, validation, recovery chain following) and (2) formatting logic (`resolveSections`, `formatAiderTrick`, `formatCheckpointForInjection`). The formatting functions (lines 305-547) are pure transformers that take a `Checkpoint` and produce strings.

**Action**:
1. Create `src/checkpoint/formatter.ts` with `resolveSections`, `formatAiderTrick`, `formatCheckpointForInjection` (+ the `CheckpointSection` interface and `LoadOptions` type if needed)
2. Update `loader.ts` to import the public API from `./formatter.js` and re-export `formatCheckpointForInjection` (to avoid breaking external consumers)
3. Update test imports if needed
4. Verify: `tsc --noEmit`, `vitest run`, `node build.ts`

**Expected result**: `loader.ts` drops to ~300 LOC, `formatter.ts` is ~250 LOC.

### WP-05: Mark remaining 3 files as wontfix

**Rationale**: These three files are barely over the threshold and are cohesive units:

- **`src/gsd/state-reader.ts` (566 LOC)**: Contains 5 tightly coupled parsing sections that all operate on the same GSD state. Splitting by "parse STATE.md" vs "parse ROADMAP.md" would create two files that both need the same regex constants and types. Net result: more files, more coupling, no clarity gain.

- **`src/hooks/session-start.ts` (521 LOC)**: Entry point hook with bootstrap + registration + main handler. The bootstrap and registration are only called from the handler. Splitting would create a helper file with functions that have exactly one caller.

- **`src/hooks/session-end.ts` (502 LOC)**: Barely 2 lines over threshold. Entry point hook with ~10 sequential sections in the handler. Each section is small (20-50 lines) and they must execute in order. No natural extraction boundary.

**Action**: For each, run:
```
desloppify resolve structural::<filepath> --status wontfix --note "Cohesive unit, barely over 500 LOC threshold. Internal structure is clean (well-sectioned). Splitting would increase coupling without improving clarity."
```
