## Analysis

The `readStdin()` function at line 25-32 calls `JSON.parse(raw)` without a local try/catch:

```typescript
export async function readStdin<T>(): Promise<T> {
  const chunks: Buffer[] = [];
  for await (const chunk of process.stdin) {
    chunks.push(chunk as Buffer);
  }
  const raw = Buffer.concat(chunks).toString('utf-8');
  return JSON.parse(raw) as T;  // line 31 — unguarded
}
```

### Risk Assessment

**Effective severity: Low (mitigated by caller)**

1. `readStdin()` is only called in one place: `runHook()` at line 54.
2. That call sits inside `runHook()`'s outer try/catch (lines 53-121), which catches the error, logs it via `logToFile()`, emits a safe fallback output via `writeStdout()`, and exits cleanly.
3. No other callers exist — `readStdin` has zero consumers outside `runHook()` (confirmed by grep).
4. The input source is Claude Code's own stdin pipe, not arbitrary user input. Malformed JSON here would indicate a Claude Code bug, not an attack vector.

So while the finding is technically valid (the function itself doesn't guard), the runtime behavior is safe because the caller always catches.

### Decision: Fix in code

Despite the low runtime risk, fixing this is trivial and makes `readStdin()` defensively correct regardless of future callers. It also clears the desloppify finding without requiring a wontfix exemption.

    chunks.push(chunk as Buffer);
